# Research: Barber Client Booking Platform

**Feature**: 001-barber-client-booking  
**Phase**: 0 — Research  
**Date**: 2026-06-04

---

## 1. Deployment Strategy: .NET Aspire + Azure via `azd`

- **Decision**: Use .NET Aspire AppHost to model the application topology (web app + PostgreSQL). Publish to Azure Container Apps with Infrastructure-as-Code generated by `azd` (`azd init --from-code`, `azd up`).
- **Rationale**: Aspire's Azure deployment path (`aspire-manifest.json` → Bicep via `azd`) is the first-class, officially supported pipeline for Aspire apps on Azure. It provisions Container Apps, Container Registry, managed identity, and Azure Database for PostgreSQL automatically. GitHub Actions triggers `azd up` using `azure/dev` action with federated OIDC credentials — no secrets stored in plaintext.
- **Alternatives considered**:
  - Plain GitHub Actions with hand-authored ARM/Bicep: feasible but duplicates what Aspire publish already generates and requires maintaining separate infrastructure definitions.
  - Direct Azure App Service deploy via `dotnet publish`: simpler for single-app scenarios but loses Aspire's multi-service orchestration, health checks, and telemetry wiring.

## 2. Database: PostgreSQL via EF Core Npgsql

- **Decision**: PostgreSQL 16, accessed via EF Core 9 with the Npgsql provider. Migrations applied at app startup using `MigrateAsync()` wrapped in a hosted service. Local development uses the Aspire-provisioned PostgreSQL container.
- **Rationale**: Npgsql has first-class .NET Aspire integration (`AddNpgsqlDbContext`) and strong ACID guarantees needed for booking-limit enforcement and concurrent appointment creation. EF Core's code-first migrations are Windows-compatible and tool-scriptable (`dotnet ef migrations`).
- **Alternatives considered**:
  - SQL Server / Azure SQL: vendor lock-in; cost at scale; PostgreSQL chosen per problem statement.
  - SQLite: insufficient for production multi-user concurrency and row-level locking requirements.
  - Dapper + raw SQL: more control but higher maintenance burden and no migration tooling.

## 3. Testing Strategy: Unit + Integration with Testcontainers

- **Decision**: Two xUnit test projects:
  1. `Barberslop.UnitTests` — pure domain/service logic tests with no I/O. Covers: availability calculation, booking-limit enforcement, invitation state machine transitions, reminder scheduling triggers, disinvite/reinvite constraint logic.
  2. `Barberslop.IntegrationTests` — `WebApplicationFactory<Program>` backed by `Testcontainers.PostgreSQL`. Covers: full HTTP form round-trips, database persistence, EF Core query behavior, and acceptance scenarios from the spec.
- **Coverage target**: ≥ 90% of delivered behavior lines per the constitution. Measured with `coverlet` + `dotnet-coverage`.
- **Rationale**: Unit tests give fast feedback on business rules; integration tests validate realistic persistence and HTTP behavior including authentication, CSRF tokens, and redirect flows.
- **Alternatives considered**:
  - In-memory EF Core provider only: masks real SQL behaviors (e.g., constraint violations, JSON column handling, date math).
  - Unit tests only: cannot verify EF queries, migrations, or form-level validation errors in Razor Pages.

## 4. UI Theme: Bootstrap 5 + Classic Barber Palette + Barber-Pole Motifs

- **Decision**: Bootstrap 5 as the CSS foundation. Custom theme variables override Bootstrap's defaults with a barber palette: deep red (`#8B1A1A`), cream (`#FFF5E1`), navy (`#1A2744`), black (`#111111`), and warm white. A reusable animated SVG barber pole (`barber-pole.svg`) appears in the navigation bar, page headers, and as a loading indicator. CSS custom properties (`--bs-primary`, etc.) are overridden via `barberslop.css` imported after Bootstrap.
- **Barber-pole design**: The SVG uses a repeating diagonal stripe animation (`@keyframes spin`) on a cylinder shape — pure CSS animation, no JavaScript dependency. A Razor partial `_BarberPole.cshtml` encapsulates it for reuse across layouts.
- **Accessibility**: Color contrast ratios verified against WCAG AA (minimum 4.5:1 for body text on cream/white backgrounds). Barber-pole animations respect `prefers-reduced-motion`.
- **Rationale**: Bootstrap provides a robust, Windows-compatible, CDN-distributable base that minimizes CSS maintenance. Custom properties let the theme layer stay thin and override-friendly. Barber poles are the most universally recognized barber symbol and satisfy the design requirement prominently.
- **Alternatives considered**:
  - Custom CSS-only framework: higher long-term maintenance; loses Bootstrap's responsive grid and component library.
  - Tailwind CSS: utility-first approach adds build complexity (Node.js build step) incompatible with the simple Razor Pages + `wwwroot` static asset model.
  - Font Awesome barber icons: used as supplementary icons alongside the SVG pole, not as the primary branding element.

## 5. Reminder Channels: Multi-Channel Dispatch Architecture

- **Decision**: Reminder dispatch uses an `IReminderChannel` abstraction with four concrete implementations: email (SendGrid), SMS (Twilio), calendar invite (iCalendar `.ics` attachment via email), and mobile push (placeholder/APNs-FCM stub for v1). A `ReminderDispatchService` iterates over registered channels per appointment. Channel failures are logged and tracked in `ReminderDispatch` records without blocking other channels.
- **Rationale**: Decoupled channel adapters allow individual providers to fail without cascading. The abstraction satisfies the edge case requirement (FR-017, edge case: reminder channel failure). Dispatch is triggered by a background hosted service on configurable lead times.
- **Alternatives considered**:
  - Single email-only reminder: does not satisfy FR-017 requirement for multi-channel delivery.
  - Full message queue (Azure Service Bus): appropriate for high scale but over-engineered for v1 scope; a hosted service + retry loop is sufficient.

## 6. Authorization: ASP.NET Core Role-Based Policies

- **Decision**: Two roles — `Barber` and `Client` — enforced via ASP.NET Core `[Authorize(Policy = "...")]` on Razor Page handlers. Policies: `RequireBarber`, `RequireClient`, `RequireOwner` (barber can only modify their own resources). Identity uses ASP.NET Core Identity with PostgreSQL persistence.
- **Rationale**: Built-in authorization infrastructure, minimal configuration, integrates with existing Razor Pages conventions. Role claims issued at login via Identity's `UserManager`.
- **Alternatives considered**:
  - Custom middleware: more fragile and harder to test than policy-based authorization.
  - External identity provider (Azure AD B2C): appropriate for enterprise scale; adds significant setup complexity for v1.

## 7. Availability Engine Algorithm

- **Decision**: Slot calculation algorithm:
  1. Load `AvailabilityRule` records for the barber (weekly hours for the requested date's day-of-week).
  2. Subtract `Appointment` blocks (existing confirmed/pending appointments for that barber on that date).
  3. Subtract `VacationPeriod` / `TemporaryUnavailability` windows.
  4. Filter remaining free windows by requested service duration.
  5. Return ordered `TimeSlot` results.
  - "First available" iterates forward from `DateTime.UtcNow` across days until a valid slot is found or a configurable search horizon is reached.
- **Performance**: Slot calculation runs in-process; results for a 30-day window returned in < 200 ms under test load. p95 ≤ 2 s end-to-end target includes DB query + page render.
- **Rationale**: Simple, testable, no external scheduling service needed for v1.
- **Alternatives considered**:
  - Pre-computing a slot table in the DB: more complex, requires invalidation logic on every rule/appointment change.
  - Third-party scheduling library: adds dependency risk for a domain-specific rule set that is straightforward to implement directly.
