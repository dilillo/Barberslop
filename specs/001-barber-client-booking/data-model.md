## Entity Overview

```text
BarberProfile ──┬── ServiceOffering (1..*)
                ├── AvailabilityRule (0..*)
                ├── VacationPeriod (0..*)
                ├── TemporaryUnavailability (0..*)
```

### InvitationRequest

Represents a client's request to join a barber's clientele, or a barber-initiated invite.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `Guid` | PK, not null | |
| `barberProfileId` | `Guid` | FK → BarberProfile, not null | |
| `clientProfileId` | `Guid` | FK → ClientProfile, not null | |
| `status` | `InvitationStatus` (enum) | not null | See state machine below |
| `initiatedBy` | `InitiatorType` (enum) | `Client` or `Barber`, not null | |
| `requestedAt` | `DateTimeOffset` | not null | |
| `decidedAt` | `DateTimeOffset` | nullable | Set when status transitions to Accepted/Rejected |
| `disinvitedAt` | `DateTimeOffset` | nullable | Set when barber disinvites |
| `disinviteReason` | `DisinviteReason` (enum) | nullable | RepeatedNoShow, LostContact, Other |
| `notificationSentAt` | `DateTimeOffset` | nullable | When acceptance notification delivered (FR-005) |

**`InvitationStatus` state machine**:

```
Pending → Accepted → Active
        → Rejected
Active  → Disinvited
```

- Only `Active` clients may book appointments.
- `Disinvited` clients cannot submit a new `InvitationRequest` (FR-006); only the barber may re-invite (FR-007) by creating a new `InvitationRequest` with `initiatedBy = Barber`.

**Invariants**:
- At most one non-`Disinvited` `InvitationRequest` may exist per (barber, client) pair.
- Transitioning to `Disinvited` requires a `disinviteReason` value.

### Appointment

Represents a scheduled booking between a barber and a client (or family member).

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `Guid` | PK, not null | |
| `barberProfileId` | `Guid` | FK → BarberProfile, not null | |
| `clientProfileId` | `Guid` | FK → ClientProfile, not null | |
| `familyMemberId` | `Guid` | FK → FamilyMember, nullable | Non-null if booked for family member |
| `serviceOfferingId` | `Guid` | FK → ServiceOffering, not null | |
| `startAt` | `DateTimeOffset` | not null | |
| `endAt` | `DateTimeOffset` | endAt > startAt, not null | Computed from startAt + service duration |
| `status` | `AppointmentStatus` (enum) | not null | See state machine below |
| `isStanding` | `bool` | default false | Recurring/standing appointment flag (FR-016) |
| `standingRecurrence` | `string` | nullable | iCalendar RRULE string if standing |
| `bookedByRole` | `BookingActorRole` (enum) | `Client` or `Barber` | Tracks who created the appointment |
| `createdAt` | `DateTimeOffset` | not null | |
| `cancelledAt` | `DateTimeOffset` | nullable | |
| `cancellationReason` | `string` | max 300, nullable | |

**`AppointmentStatus` state machine**:

```
Pending → Confirmed → Completed
        → Cancelled
Confirmed → Cancelled
```

**Invariants**:
- On creation: barber must have an `Active` `InvitationRequest` with the client.
- On creation: `startAt` must fall within a valid availability slot (no overlap with existing appointments, vacation, or temporary unavailability).
- On creation: the client's count of future `Confirmed` appointments with this barber must be < `barber.defaultBookingLimit` (FR-014).
- On `VacationPeriod`/`TemporaryUnavailability` creation: if a `Confirmed` appointment conflicts, a separate resolution flow is triggered (edge case in spec).

### ReminderDispatch

Tracks the state of a reminder sent (or to be sent) for a specific appointment on a specific channel.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `Guid` | PK, not null | |
| `appointmentId` | `Guid` | FK → Appointment, not null | |
| `channel` | `ReminderChannel` (enum) | Email, SMS, CalendarInvite, Push | |
| `scheduledFor` | `DateTimeOffset` | not null | When reminder should be dispatched |
| `status` | `ReminderStatus` (enum) | Pending, Sent, Failed | |
| `attemptedAt` | `DateTimeOffset` | nullable | |
| `failureReason` | `string` | max 500, nullable | |
| `externalMessageId` | `string` | max 255, nullable | Provider-assigned ID (e.g., SendGrid message ID) |

**Invariants**:
- One `ReminderDispatch` row per (appointment, channel, scheduledFor) combination.
- `Failed` dispatches are logged but do not block other channel dispatches.

## Key Validation Rules

| Rule | Entities | Validation |
|---|---|---|
| Booking limit | `Appointment`, `BarberProfile` | COUNT of future `Confirmed` appointments for (barber, client) < `defaultBookingLimit` |
| No double-booking | `Appointment` | No overlap of `[startAt, endAt)` for same barber |
| Availability window | `Appointment` | `startAt` must fall in computed available slot |
| Vacation conflict | `Appointment`, `VacationPeriod` | Appointment must not overlap vacation period |
| Disinvite block | `InvitationRequest` | `Disinvited` client cannot create new `InvitationRequest` |
| Active relationship required | `Appointment` | `InvitationRequest.status == Active` required before booking |
| Family member ownership | `Appointment`, `FamilyMember` | `familyMember.clientProfileId == appointment.clientProfileId` |
| Service active | `Appointment`, `ServiceOffering` | `serviceOffering.isActive == true` at booking time |
| Availability rule non-overlap | `AvailabilityRule` | Multiple rules for same day must not have overlapping time windows |
